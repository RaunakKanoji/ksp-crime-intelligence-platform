# Feature Audit

Status date: 2026-07-11

This audit covers specifications `001` through `060` and the current Next.js/Catalyst implementation. The specifications are present under `context/docs/features/`. The implementation is broad, but most production-facing workflows still use deterministic sample data, local/browser state, in-memory globals, or mock aggregation services. That means this phase is not complete yet; this document is the baseline for controlled database migration.

## Global Findings

- All 60 specifications have a corresponding spec file.
- Most features have a route, component, and service/API module.
- Catalyst Authentication is partially integrated client-side for login/signup, with a localhost demo-session fallback.
- Phase 1 database artifacts and server persistence wiring now exist for users, roles, user-role assignments, jurisdiction access, and audit events. They are not live until Catalyst Console setup/imports and server adapter/session configuration are complete.
- Persistent Catalyst Data Store access is not yet implemented for FIR/search/map/analytics feature data.
- Permission checks exist through `src/lib/permissions.ts`, but the matrix can be modified through `localStorage` and server globals, so it is not an authoritative production permission source.
- Many pages expose sample/demo/mock data in production-facing routes.
- The main navigation currently exposes nearly every feature as a primary item, producing clutter and competing workflows.
- API response shapes and error handling vary by feature.
- Several routes are duplicate entry points for similar concepts: FIR search vs advanced filters, map vs crime-map, AI query vs explanation/history, analytics sub-pages, and admin/data operation pages.

## Mock/Data Usage Classification

| Pattern | Classification | Action |
|---|---|---|
| `src/lib/crime-map/mock-crime-data.ts` and imports | Temporary mock backing many production pages | Replace after `crime_records`, `locations`, and `jurisdiction_boundaries` repositories exist |
| `source: "mock"` response fields | Temporary mock marker | Retain only for demo mode or development fixtures |
| `setTimeout` in services | Simulated network latency | Remove during repository migration; keep only UI auto-dismiss timers |
| `localStorage` saved FIR filters | Temporary client preference | Replace with `saved_views`/`saved_queries`; local cache can remain convenience-only |
| `localStorage` permission matrix/demo role | Development-only | Remove as authority source for production |
| Domain `data.ts` files | Temporary fixtures | Move to seed/test fixture area or replace with datastore reads |
| Demo mode reset of local keys | Legitimate development-only | Keep behind demo permission and production guard |
